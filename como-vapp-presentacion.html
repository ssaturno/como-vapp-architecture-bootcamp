01 · Propósito del Sistema
Un sistema de pedidos
cloud-native y observable
Plataforma web que permite a usuarios hacer seguimiento de pedidos, y a clientes (negocios) gestionar y controlar el estado de las órdenes.
  • ✓Toma y validación de pedidos — el usuario crea una orden y queda persistida en DynamoDB.
  • ✓Seguimiento de estados — el cliente actualiza el estado y el sistema notifica automáticamente.
  • ✓Notificación por email — cada cambio de estado dispara un correo via SES.
50 pedidos/día ~1 pedido/min pico SLA < 2 min notificación
Actores del sistema
👤
Usuario Final
Crea pedidos y consulta su estado desde el frontend React. No tiene cuenta — solo ID de pedido.
🏪
Cliente (Negocio)
Usa el panel de administración (Kotlin) para gestionar y cambiar estados de pedidos.
MVP v1 — Exclusiones
❌ Pagos en plataforma  ·  ❌ Tracking por mapas  ·  ❌ Cuentas de usuario

Infraestructura con Terraform

Provisioning de VPC, DynamoDB, SQS, Lambda, ECR, S3 y más

03 · Terraform
¿Qué provisiona Terraform?
~35–40 recursos de AWS. El clúster EKS se crea aparte con eksctl.
🌐
vpc.tf
VPC + subnets públicas/privadas + IGW + NAT Gateway + VPC Endpoints (DynamoDB, S3)
🗄️
dynamodb.tf
Tabla orders · PAY_PER_REQUEST · Streams NEW_AND_OLD · TTL · PITR · SSE
📨
sqs.tf + ses.tf
Cola de notificaciones + Dead Letter Queue + email identity SES verificado
λ
lambda.tf + iam.tf
Función Python que lee DynamoDB Streams y publica en SQS. Usa LabRole en Academy.
📦
ecr.tf + s3.tf
4 repos ECR (scan on push) + bucket S3 de frontend estático + bucket AWS Config
📊
cloudwatch.tf + config.tf
Log groups (30 días) · Alarmas SQS/Lambda · AWS Config recorder + reglas de compliance
03 · Terraform — Flujo
Pasos para aplicar
la infraestructura
Desde cero hasta infraestructura lista para recibir el clúster EKS.
1
Verificar email SES
aws ses verify-email-identity --email-address tu@email.com
2
terraform init + plan
cd infra/terraform && terraform init && terraform plan -out=tfplan
3
terraform apply
Crea ~35-40 recursos. Guardar outputs con terraform output -json

Usar outputs en K8s
Reemplazar placeholders en ConfigMaps con URIs de ECR, tabla DynamoDB, URL SQS

Misión 2 — Seguridad

IAM · Secrets Manager · Cgroups · Seccomp · AppArmor · RBAC · Trivy

05 · Misión 2 — Seguridad
Seguridad por capas — Defense in Depth
Cada capa protege la siguiente. Si una falla, las demás siguen en pie.
🔑
Secretos & Credenciales
AWS Secrets Manager + Secrets Store CSI Driver. Montados en /mnt/secrets/. Nunca en variables de entorno ni en el código.
🛡️
IAM & Mínimo Privilegio
Cada servicio tiene exactamente los permisos que necesita. IRSA por pod en producción. En Academy: LabRole compartido.
🔒
Cifrado
DynamoDB SSE con AWS-managed key. SQS cifrado. Tráfico en tránsito con TLS. S3 solo HTTPS.
📦
Container Hardening
Non-root (UID 1001) · readOnlyRootFilesystem · capabilities: drop ALL · Seccomp RuntimeDefault · AppArmor
🔍
Escaneo de Imágenes
SonarQube en el pipeline (código fuente) + Trivy (imágenes Docker). Pipeline bloqueado si hay vuln. críticas.
🌐
Red & Perímetro
WAF en ALB. Pods en subnets privadas sin IP pública. VPC Endpoints para DynamoDB y S3. Session Manager en lugar de SSH.
05 · Secretos & Configuración
Gestión de secretos
sin credenciales en código
Tres capas según el tipo de dato y su sensibilidad.
DatoAlmacenamientoInyección
Credenciales BD, API Keys, tokens AWS Secrets Manager CSI Driver → /mnt/secrets/
Config pública: URLs, puertos, región ConfigMaps Variables de entorno en Deployment
.env locales (solo desarrollo) Archivos locales .gitignore — nunca en git
Rotación centralizada Sin credenciales en logs KMS en reposo
Flujo de montaje de secretos
1
Secretos en AWS Secrets Manager Credenciales de DynamoDB, SES, SQS — cifradas con KMS
2
Secrets Store CSI Driver Integración nativa EKS → Secrets Manager
3
Montaje en tiempo de ejecución Pod lee secretos desde /mnt/secrets/ del filesystem
4
Rotación automática Cambio en Secrets Manager → se refleja en el próximo ciclo de montaje

Separación de ambientes y mínimo privilegio dentro del clúster.
🗂️

como-vapp-dev y como-vapp-prod completamente aislados. RBAC garantiza que no se puedan hacer cambios en Prod desde Dev.
Dev Prod
👤
ServiceAccount por servicio
orders-sa, admin-sa, notifications-sa — cada pod tiene su propia identidad. Preparados para IRSA en producción real.

Observabilidad & Diagnóstico

CloudWatch · Prometheus · Grafana · Liveness & Readiness Probes

06 · Observabilidad
Stack de monitoreo: CloudWatch + Prometheus + Grafana
Cada capa cubre lo que la otra no ve de forma granular.
☁️
Amazon CloudWatch
Logs de DynamoDB, SQS, SES, Lambda — nativo de AWS sin configuración. Log groups con retención de 30 días. Alarmas automáticas.
📡
Prometheus
Métricas granulares de pods EKS: latencia de endpoints, tasa de errores HTTP, uso de memoria por pod. Instalado con Helm.
📈
Grafana
Dashboards: réplicas activas · CPU/RAM por pod · tasa de errores HTTP · longitud de cola SQS · pedidos últimas 24h.
Logs estructurados JSON — campos en cada registro
timestamp service severity requestId userId endpoint-action message description
06 · Probes & Diagnóstico
Liveness & Readiness
— salud del runtime
Sin probes, K8s no puede distinguir un pod roto de uno sano.
Liveness Probe
Le dice a K8s si el pod está vivo. Si falla repetidamente → reinicio automático. HTTP GET /health en cada servicio.
Readiness Probe
Le dice a K8s si el pod está listo para tráfico. Si falla → se saca del load balancer hasta que se recupere. Evita enviar requests a pods dañados.
Comandos de diagnóstico
kubectl -n como-vapp-dev get pods
kubectl -n como-vapp-dev describe pod <pod-name>
kubectl -n como-vapp-dev logs -f deploy/orders-service
kubectl -n como-vapp-dev get events --sort-by=.metadata.creationTimestamp
Flujo de diagnóstico ante fallo
1
Alerta automática — CloudWatch Alarms detecta anomalía
2
Identificar componente — Grafana dashboards / kubectl get pods
3
Consultar métricas — Prometheus + Grafana: CPU, RAM, latencia, errores
4
Revisar logs — CloudWatch Logs / Logs Insights → causa raíz
5
Verificar eventos del clúster — kubectl describe / CrashLoopBackOff, OOMKilled
✓
Corregir y validar — rollout restart / rollback → confirmar en Grafana
06 · Pruebas de Resiliencia
Plan de pruebas controladas
Simulamos fallos reales en staging para validar auto-recuperación y observabilidad antes de producción.
💥
Stress Testing — CPU/Memoria
Comando: stress-ng --cpu 2 --vm 1 --vm-bytes 512M
Objetivo: validar cgroups, throttling sin crash, auto-recuperación.
🗑️
Fallos de Contenedores
Comando: kubectl delete pod <pod>
Objetivo: validar self-healing de K8s y funcionamiento de probes.
🔌
Fallos de Conectividad
Simular caída del servicio de notificaciones. Mensajes deben quedar en SQS y procesarse al recuperarse el pod.
🔄
Rollback a versión anterior
kubectl rollout history → kubectl rollout undo
Objetivo: confirmar que cualquier despliegue es reversible sin downtime.

Evidencias concretas que demostraremos en la presentación final.

  • Error inyectado → logs estructurados → causa raíz
  • Métricas: CPU, requests/min, errores HTTP, SQS queue depth
✓ Infraestructura
  • terraform apply sin errores (~35-40 recursos)
  • eksctl create cluster funcionando
  • Imágenes en ECR con tag semántico
  • kubectl get pods → todos Running/Ready
